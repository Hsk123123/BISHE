## 二、业务流程验证

### 完整预约服务流程

**测试账号**：买家账号（role=0），服务者账号（role=1），管理员账号（role=2）

**前置条件**：
- 服务者已发布技能并通过审核
- 服务者已在日程管理中添加 2026-04-25 上午时间段
- 买家钱包余额 ≥ 技能价格

| 步骤 | 操作 | 订单状态 | 钱包变化 | 时间段状态 |
|------|------|----------|----------|------------|
| 1 | 买家浏览技能，查看详情，自动记录浏览历史 | — | — | 空闲(0) |
| 2 | 选择 2026-04-25，选择"上午(09:00-12:00)" | — | — | 空闲(0) |
| 3 | 点击"确认预约"，弹窗确认 | 待支付(0) | 未变化 | 锁定(2) |
| 4 | 订单页点击"支付" | 待接单(1) | 余额 -N 元，写流水 | 已预约(1) |
| 5 | 服务者端"技能订单"看到新订单，点击"接单" | 已接单(2) | — | — |
| 6 | 服务者点击"开始服务" | 服务中(3) | — | — |
| 7 | 服务者点击"完成服务" | 待评价(4) | — | — |
| 8 | 买家提交评价（评分 + 内容） | 已完成(5) | 积分 +N | — |

**验证结论**：8 步流程全部走通，各状态跳转与数据库一致，时间段防重复预约机制正常。

## 三、异常与边界测试

| 编号 | 测试场景 | 触发方式 | 后端响应 | 前端展示 | 结果 |
|------|----------|----------|----------|----------|------|
| E-01 | 余额不足时支付 | 钱包余额 0，点击支付 | HTTP 400，message: "余额不足" | Toast 提示"余额不足" | 通过 |
| E-02 | 购买自己发布的技能 | 服务者账号进入自己的技能详情，尝试下单 | HTTP 400，message: "不能购买自己的技能" | Toast 提示错误信息 | 通过 |
| E-03 | 重复添加相同时间段 | 排班管理添加已存在的 日期+时间段 | HTTP 400，message: "该时间段已被占用" | Toast 提示错误信息 | 通过 |
| E-04 | 删除已预约时间段 | 直接调用 DELETE /schedule/{id}，该时间段 status=1 | HTTP 400，message: "只能删除空闲状态的时间段" | Toast 提示错误信息 | 通过 |
| E-05 | 删除不属于自己的时间段 | 直接调用 DELETE /schedule/{id}，非本人 | HTTP 400，message: "无权操作" | Toast 提示错误信息 | 通过 |
| E-06 | 添加过去日期的时间段 | 选择历史日期提交 | HTTP 400，message: "不能添加过去的时间段" | Toast 提示错误信息 | 通过 |
| E-07 | Token 过期后访问接口 | JWT 过期后操作任意需登录接口 | HTTP 401 | 自动跳转登录页，提示"登录已过期" | 通过 |
| E-08 | 普通用户访问管理员接口 | role=0 直接请求 /admin/** | HTTP 403 | 路由守卫拦截，重定向至首页 | 通过 |
| E-09 | 非法状态跳转（重复接单） | 订单已接单，再次调用接单接口 | HTTP 400，状态机拒绝 | Toast 提示操作非法 | 通过 |
| E-10 | 重复提交评价 | 已完成订单再次提交评价 | HTTP 400，message: "订单状态不允许评价" | Toast 提示错误信息 | 通过 |

## 五、测试结论

**系统整体状态：稳定可用**

| 维度 | 结论 |
|------|------|
| 功能完整性 | 核心功能全部实现，用户端、服务者端、管理后台三端均可正常运行 |
| 业务闭环 | 浏览→预约→支付→服务→评价全链路跑通，无断点 |
| 数据一致性 | 支付、退款、状态变更均在事务内执行，无脏数据 |
| 异常处理 | 10 类边界场景均返回明确错误信息，前端有对应提示 |
| 安全性 | JWT 鉴权覆盖全部需登录接口，角色校验防止越权操作 |
| 排班系统 | 时间段防重复、防非法删除逻辑均验证通过 |

**存在的已知边界**：
- 支付为平台内部钱包扣款，不对接第三方支付渠道
- 文件上传依赖本地或 MinIO 配置，演示环境使用默认图片
- 签到积分记录存数据库，无 Redis 缓存层
